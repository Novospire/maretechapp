# Design Reference — Maretech

## Purpose

This file keeps Maretech visual/design direction separate from backend, AI pipeline and product logic.

Use this file before UI, layout or marketing site work.

## Product Feeling

Maretech should feel:

* technical
* marine-focused
* trustworthy
* calm
* precise
* non-alarmist
* professional

It should not feel:

* like a medical diagnosis app
* like a legal survey/certification tool
* like a generic AI gimmick
* like a speculative crypto/finance product

## App vs Marketing Site

Maretech App:

* guided capture
* visual signal tracking
* timeline/compare
* decision support
* monitor / re-check / escalate flow

Maretech Marketing Site:

* explain the product
* build trust
* show workflows
* explain boundaries
* convert users

Do not mix app implementation and marketing site design in one ticket.

## Visual Language

Prefer:

* marine-tech clarity
* structured cards
* clear capture steps
* restrained confidence/risk indicators
* clean dashboard/timeline patterns
* sober blue/navy/white/gray palette

Avoid:

* panic colors as primary UI
* “AI magic” visuals
* definitive diagnosis badges
* certification-style seals
* surveyor-report language

## Allowed UI Work

* layout refinement
* visual hierarchy
* copy clarity
* onboarding screens
* capture guidance UI
* result presentation UI
* marketing landing sections

## Forbidden UI Work Unless Approved

* changing product flows
* changing pricing logic
* adding diagnosis language
* merging Osmosis and Corrosion flows
* backend/API changes
* database changes
* deployment changes

## Hard Rule

Design must reinforce Maretech as visual-signal decision support, not as diagnosis, certification or survey replacement.
